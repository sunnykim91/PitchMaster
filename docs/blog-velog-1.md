# 축구팀 회장 4년 하다가 직접 서비스 만든 이야기

> PitchMaster 개발기 시리즈 1편 — 왜 만들었나

매주 카톡 공지방에 "이번 주 참석 투표 부탁드려요" 올리는 거. 해본 사람은 압니다. 그게 얼마나 지치는 일인지.

저는 평일 저녁 축구팀을 4년째 운영하고 있어요. 회원 37명, 매주 경기. 축구 자체는 재밌습니다. 근데 경기 말고 귀찮은 게 너무 많았어요.

---

## 매주 반복되던 세 가지

### 1. 참석 투표 — 갠톡으로 끝나는 매주

카톡 공지방에 투표 글 하나 올려놓습니다. 한 번 올리면 끝일 것 같죠? **안 하는 사람은 네 번을 올려도 안 합니다.** 공지방이 따로 있는데도요.

결국 갠톡합니다. 37명 중 매주 5~6명은 꼭 갠톡으로 답을 받아야 했어요. **이걸 매주 4년간** 했습니다.

### 2. 회비 정산 — 카뱅 모임통장 + 엑셀

카뱅 모임통장을 매달 열어요. 누가 넣었는지 한 명씩 보고 엑셀에 정리합니다. 이름 대조하고, 안 낸 사람 추리고, 따로 연락하고. 별것 아닌 것 같은데 매번 30분 가까이 걸렸어요.

### 3. 라인업 — 제일 머리 아픈 것

15명 와서 4쿼터 나눠야 하는데:

- 선호 포지션이 다 다름 (공격만 / 수비도 OK / GK 가능 등)
- 출전 시간을 골고루 맞춰야 함
- 용병 2~3명 끼면 더 복잡
- 전반만 뛰는 사람 / 후반만 뛰는 사람도 처리

**나름 신경 써서 짜도 "나 오늘 왜 수비만 해?" 한 마디 꼭 나옵니다.**

이 세 가지가 사실상 팀 운영의 전부예요. 구장 예약·경기 잡는 건 한 번 하면 끝이지만, 매주 머리 쓰는 건 결국 저 셋.

---

## 찾아봤는데 진짜 없었어요

매주 반복되는 거니까 누가 만들어놨겠지 싶어서 진지하게 찾아봤습니다. 밴드, 스포츠매치, 게임원, 플코, 바로매치, 온더그라운드... 앱스토어에 풋살 키워드로 다 깔아봤어요.

근데 **딱 맞는 게 없었어요.**

- 투표는 되는데 회비 관리 X
- 회비 관리는 되는데 라인업 X
- 라인업 자동 배치는 아예 없음
- 매칭 앱은 많은데 우리는 이미 같은 팀이라 매칭이 필요 없음

조기축구·풋살 팀을 **실제로 운영하는 총무 입장에서** 매주 겪는 이 문제를 풀어주는 서비스가 없었습니다. 그냥 직접 만들기로 했어요.

---

## 기술 스택은 빨리 쓸 수 있는 걸로

사이드 프로젝트라 거창하게 갈 생각 없었어요. 기준은 하나: **"이번 주 경기 전에 쓸 수 있느냐"**.

```
Frontend / Backend: Next.js (App Router)
Database:          Supabase (PostgreSQL + RLS)
Hosting:           Vercel (git push 자동 배포)
Auth:              Kakao OAuth
PWA:               Service Worker + Web Push
```

**Next.js**: 프론트랑 백엔드를 한 프로젝트에서. 서버 따로 띄울 필요 X. 혼자 할 때 부담 적음.

**Supabase**: PostgreSQL이라 회비 같은 돈 데이터 다루기 편함. 무료 티어 시작 가능. 나중에 RLS 정책으로 팀별 데이터 격리.

**Vercel**: 무료 + git push 자동 배포. 도메인만 붙이면 끝.

**카카오 로그인 (필수)**: 이메일 회원가입 넣으면 팀원들한테 "이거 써봐" 했을 때 회원가입 화면에서 다 이탈할 게 뻔했어요. 어차피 팀 운영이 카톡으로 굴러가니까 카카오 로그인이 가장 자연스러웠습니다.

---

## 우리 팀부터 — 1주일 적용

처음엔 완전히 우리 팀 전용으로 만들었어요. 팀 이름도 하드코딩이었습니다 ㅋㅋ

기능 3개만 넣고 시작했어요:
- 참석 투표
- 회비 관리
- 경기 기록

그동안 쌓인 데이터(과거 경기·회비·회원) 전부 옮겼어요. **1주일 동안 실제로 우리 팀에 적용**하면서 매일 불편한 거 하나씩 고쳤습니다.

팀원 반응? "이거 편하다"보다는 **"이제 공지방에 투표 네 번 안 올려도 되냐"** 에 가까웠는데, 어쨌든 쓰기 시작했어요. 링크 하나면 끝이니까 투표율은 확 올랐습니다.

---

## 다른 팀도 쓸 수 있게 — 사용성 갈아엎기

우리 팀에서 잘 굴러가니 다른 팀도 받아보고 싶어졌어요. 근데 단순 "팀 추가" 기능만 넣으면 안 됐습니다.

> 우리 팀은 제가 만들어서 좀 불편해도 씁니다. 다른 팀은 그렇지 않잖아요. 처음 들어와서 몇 분 안에 "괜찮은데?" 가 안 들면 그냥 닫아버립니다.

그래서 디자인·사용성에 시간을 많이 썼어요:
- 가입 시 묻는 정보 최대한 줄임
- 한 화면에서 가능한 액션 늘림
- 버튼 한 번에 끝나게

**큰 기능 한 번에 X. 작은 불편 매일 하나씩 ✓.** 출근 전에 하나, 퇴근 후에 하나. 그게 쌓이니 꽤 달라지더라고요.

---

## 나만 불편한 게 아니었구나

어느 정도 모양 갖춰진 후 네이버 카페·다음 카페에 글 올려봤어요. 솔직히 반응 없을 거라 생각했습니다.

근데:

> "우리 팀 총무한테 알려줘야겠다"
> "이런 거 딱 찾고 있었는데"
> "이거 만든 분 진짜 천재"

댓글이 달리기 시작했어요. **"아 이게 나만 불편했던 게 아니었구나"** 그 느낌이 묘했습니다.

지금은 **89개 팀, 700+ 명 회원**이 쓰고 있어요. 이 과정은 3편에서 자세히 풀어볼 예정입니다.

---

## 매주 쓰니까 계속 만들게 됩니다

사이드 프로젝트 몇 개 해봤는데 이것만 유독 오래 갑니다. 이유는 단순해요.

**제가 매주 씁니다.**

매주 경기 전 직접 투표 링크 보내고, 회비 정리하고, 라인업 버튼 누릅니다. 그러다 "이 버튼 위치 좀 애매한데" "여기서 저 정보가 같이 보이면 좋겠는데" 같은 게 자연스럽게 쌓여요. 바로 고치고, 다음 주에 확인.

만드는 사람이 쓰는 사람이니까 **"이 기능 진짜 필요한가?" 같은 고민을 안 해도 돼요.** 필요하면 만들고, 아니면 안 만들고.

사이드 프로젝트가 흐지부지 안 되는 이유, 결국 이거 같아요. **내가 쓸 게 있으니까.**

## 4월 24일 — 푸시가 폭탄으로 나갔습니다

MVP 투표 시작 알림 푸시 기능을 추가했어요. 컬럼 하나 만들고 (`mvp_push_sent_at`), "NULL 인 완료 경기 찾아서 푸시 보내고 sent_at 채우는" 크론 패턴이었습니다.

배포하고 한참 안 돼서 사용자들한테 수개월 전 경기에 대한 MVP 투표 알림이 가버렸어요. 컬럼은 새로 추가됐는데 **기존 완료 경기 row가 전부 NULL** 이었거든요. 크론이 다 미발송으로 잡고 한 번에 다 보낸 거였어요. 마이그레이션에 백필 한 줄 빼먹은 결과였습니다.

긴급 SQL로 일단 막았어요.

```sql
UPDATE matches
SET mvp_push_sent_at = NOW()
WHERE status = 'COMPLETED' AND mvp_push_sent_at IS NULL;
```

그리고 마이그레이션 자체에 백필을 박아 다시 배포. 그 후로 비슷한 패턴 (`*_sent_at` IS NULL 크론) 만들 때마다 백필을 무조건 같이 넣게 됐어요. 다음에 추가한 역할 가이드 푸시(`role_guide_push_sent_at`)는 처음부터 백필 포함이라 사고 안 났습니다.

푸시 알림은 한 번 잘못 나가면 사용자한테 진짜 미안해지는 종류의 사고예요. 이게 한 번 나간 후로 푸시 관련 작업할 때 더 신중해지게 됐습니다.

## 4월 28일 — redirect 한 줄로 라이브가 다운됐어요

이건 진짜 어이없는 사고였습니다.

SEO 정리하다가 `next.config.ts`에 `www → non-www` redirect 한 줄을 추가했어요. 별거 아닌 1줄.

```ts
async redirects() {
  return [{
    source: "/:path*",
    has: [{ type: "host", value: "www.pitch-master.app" }],
    destination: "https://pitch-master.app/:path*",
    permanent: true,
  }];
}
```

배포 직후 사이트가 안 열렸어요. `ERR_TOO_MANY_REDIRECTS`. 대문 그냥 닫힘.

원인은 단순한데 미리 못 봤어요. CLAUDE.md에 "Cloudflare DNS + Vercel 자동배포" 라고 본인 손으로 적어뒀는데도 까먹었거든요.

Vercel 도메인 설정에서 이미 www → non-www redirect 처리하고 있었고, Cloudflare에도 Page Rule이 있었어요. 거기에 코드 단에서 한 번 더 추가하니까 두 시스템이 핑퐁 친 거예요. A가 "non-www로 가" 보내면 B가 "www로 가" 보내고 다시 A가 ... 무한루프.

3분 만에 발견하고 revert 푸시.

```
225187a — www redirect 추가 (사고)
963b725 — 즉시 revert (복구)
```

3분이긴 한데 그 시간 동안 사이트가 진짜 안 열렸어요. 인스타 광고 진행 중인 시점이라 광고비 일부가 그냥 날아갔습니다.

그 후로 "인프라 단에서 이미 처리하고 있는 건 코드에서 절대 안 만진다"는 게 머리에 박혔어요. localhost에선 절대 재현 안 되는 사고라서요.

## 4월 29일 — Supabase가 86%를 잡아먹었어요

이건 좀 무서운 사고였어요.

경기 목록 화면에서 다른 사람 출석 투표가 실시간 반영되면 좋겠다 싶어서 Supabase Realtime 구독을 넣었어요. `match_attendance` 테이블 변경을 받아오게 했습니다.

```ts
.on('postgres_changes', {
  event: '*',
  schema: 'public',
  table: 'match_attendance',  // ← filter 안 넣음
}, ...)
```

filter를 안 넣었어요. 개발 환경에선 데이터 적어서 문제 없었거든요.

배포 후 Supabase 대시보드 들어갔다가 깜짝. **WAL 폴링이 전체 DB CPU 시간의 86%를 점유** 하고 있더라고요.

당연한 거였어요. 700명 회원이 매주 출석 투표 누르면 분당 수십~수백 변경이 일어나는데, 그걸 모든 클라이언트가 다 받아 처리하라고 한 거니까요.

해당 구독 즉시 제거. 대신 본인 액션 후 refetch하거나 같은 탭 안에선 window CustomEvent로 동기화하는 패턴으로 바꿨습니다. Realtime이 진짜 필요한 곳(경기 상세 페이지)에만 `match_id=eq.${matchId}` filter 박아서 사용.

이거 겪고 나서 Realtime 진짜 신중하게 쓰게 됐어요. "실시간이면 좋겠다"는 욕심이 부른 사고였습니다.

## 5월 2일 — 누가 SQL 인젝션 시도했어요

운영 중에 어드민 페이지에서 의심스러운 가입자가 보였어요. 카카오 ID 4875582850, 닉네임이

> `'; DROP TABLE users; --`

진짜 그렇게 적혀있었습니다. 봤을 때 좀 웃겼는데, 누가 진짜 시도하는구나 싶어서 동시에 무서웠어요.

Supabase는 prepared statement 쓰니까 직접 SQL 실행은 안 되지만, 닉네임이 그대로 DB에 저장되면 표시 단에서 다른 우회가 가능할 수 있어요. 그래서 즉시 입력 검증 헬퍼 추가했습니다.

```ts
const FORBIDDEN = /['";\\<>\-]|--|\/\*|\*\//;
```

근데 이게 진짜 황당한 게, 1차 보강(커밋 ca071e3) 배포하고 **7분 만에 같은 카카오 ID로 재가입 + 자모 우회 성공** 했어요.

```
ㅁㅁㅁㅁㅁㅁㄴㄹㅁㄴㅇㄹ
```

자모만 있는 닉네임은 1차 보강에서 안 막혔거든요. 팀명 검증에는 "의미 있는 글자 필수" 옵션을 넣었는데 닉네임 검증에는 안 넣었어요.

7분 만에 우회당한 거 보고 좀 식겁했어요. 누군가 진짜 자동화 도구로 이걸 모니터링하고 있는 거잖아요.

2차 보강에서 닉네임 진입점 6곳 다 점검해서 같은 검증 박았어요. 그리고 차단된 카카오 ID는 다음에 같은 ID로 재가입 시도해도 못 들어오게 흐름 추가:

```ts
const { data: blocked } = await db
  .from("users")
  .select("id, deleted_at, is_banned")
  .eq("kakao_id", kakaoProfile.id)
  .or("deleted_at.not.is.null,is_banned.eq.true")
  .maybeSingle();
if (blocked) throw new Error("ACCOUNT_BLOCKED");
```

이걸 겪고 나서 "검증은 한 곳만 막으면 의미 없다"가 진짜 와닿았어요. 모든 진입점에 일괄 적용해야 합니다.

## 작은 사고 하나 더 — 풋살팀에 축구 옵션이 보였어요

위 4건 외에 작은 게 하나 더 있었습니다. 설정 페이지에서 풋살팀 사용자가 "왜 11인제 옵션이 보이지?" 라고 피드백 줬어요.

`useApi(url, fallback, { skip })` 패턴을 쓰는데, fallback 객체에 `sport_type` 필드를 빼먹었거든요. skip 조건이 true 라 fetch 안 일어나고 fallback이 그대로 data로 쓰였는데, undefined가 SOCCER 기본값으로 떨어진 거였어요.

별 거 아닌 거 같은데 사용자 입장에선 "왜 우리 팀에 11인제 옵션이 보이지?" 싶었을 거예요. 1·2차 fix가 SSR 경로만 봐서 두 번 재발했다가, 사용자 콘솔 로그로 진짜 원인 확정해서 fix.

`useApi` fallback 객체는 그 후로 항상 API 응답 타입의 모든 필드 채우게 됐어요.

## 사고 후에 따라온 것들

위 5건 다 영역이 달라요. 푸시·인프라·DB·보안·프론트. 공통점은 다 미리 못 본 사각지대에서 났다는 거예요.

그래서 사고 처리하고 나면 그냥 넘기지 말고 매번 두 가지를 했습니다.

하나는 마크다운 메모에 박아두는 거예요. 어떤 증상이었고 원인이 뭐였고 해결을 어떻게 했고 다음에 비슷한 패턴 보면 어디 점검해야 하는지. 이걸 grep 가능한 형태로 누적하니까 다음 작업할 때 "이거 비슷한 사고 있었나?" 하고 한 번 찾아보게 되더라고요.

또 하나는 그 사고 케이스에 대한 테스트를 추가하는 거예요. 처음엔 0개였는데 지금은 800개 좀 넘어요. 모든 코드를 100% 커버한 게 아니라 한 번이라도 깨졌던 부분만요. 이게 누적되니까 Next.js 16 적용도, React 19 마이그레이션도 거의 무사고로 넘어갈 수 있었어요.

```bash
npx vitest run
# 800+ tests
```

이 한 줄 통과하는 거 보면 좀 안심됩니다.

## CS는 채널 두 개

처음엔 채널 여러 개 열어두려다가 정리했어요. 지금은 두 개만 씁니다.

앱 안에 "피드백 보내기" 버튼이 mailto로 제 메일(`tjsgnl2002@gmail.com`)로 직접 와요. 두 번째는 오픈카톡방 — 사용자가 직접 들어와서 질문하는 곳이고요.

질문이 자주 반복되면 가이드 페이지에 추가합니다. 사용자가 묻기 전에 답이 보이게요.

기능 요청 거절할 때는 그냥 솔직히 말합니다. "지금 자원으로는 못 만들겠어요" 또는 "그건 다른 방향으로 가고 있어서 어려울 것 같아요". 의외로 사용자들이 솔직한 거에 반감이 없더라고요.

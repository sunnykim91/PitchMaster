## 맨 위 소개 (수정)

6년간 디자인 시스템 툴링, 키오스크, CMS 등 다양한 프로덕트를 처음부터 설계하고 운영해왔습니다.

주어진 스펙을 구현하는 것에서 멈추지 않고, 실제 사용자가 겪는 불편함을 찾아 기술로 해결하는 데 관심이 많습니다. 직접 4년간 운영한 풋살 팀의 비효율을 없애겠다고 시작한 사이드 프로젝트가 현재 42개 팀 · 180명 이상이 실사용하는 서비스로 성장했고, 그 과정에서 OCR 파싱, SSR 최적화, N+1 쿼리 제거, PWA 웹 푸시까지 풀스택으로 문제를 해결해왔습니다.

팀 안에서도 반복되는 작업을 줄이는 구조나 도구를 자연스럽게 만들어온 편이고, 개발자로서 제품 전체를 함께 고민하는 걸 즐깁니다.

## PitchMaster 섹션 (수정)

### PitchMaster — 조기축구·풋살 팀 관리 플랫폼
2026.02 - 현재 | 1인 기획·개발·운영

실제 서비스 : https://www.pitch-master.app

직접 4년간 운영한 팀의 비효율(카톡 읽씹, 엑셀 회비 정산, 현장 라인업)을 해결하려고 만든 풀스택 웹앱.
42개 팀 · 180+ 회원이 실사용 중이며, API 53개 · DB 마이그레이션 11개 · 테스트 615개 규모.

**[문제 해결] 은행별 비정형 OCR 파싱**
통장 거래내역 스크린샷/엑셀의 은행마다 다른 포맷을 처리해야 했습니다. Naver Cloud OCR API로 텍스트를 추출한 뒤, 날짜·금액·입금자를 정규식 + 퍼지 매칭으로 파싱하고 팀원 이름에 자동 매칭하는 파이프라인을 구축했습니다. 한 자리 월/일(3.27), 콤마 포함 금액, 중복 거래 필터링까지 처리해 월 30분 수기 작업을 30초로 단축했습니다.

**[문제 해결] SSR↔CSR 데이터 불일치 해결**
서버에서 prefetch한 데이터와 클라이언트 hydration 시점의 데이터가 달라 React hydration error가 반복 발생했습니다. new Date()의 서버/클라이언트 시간차가 원인이었고, 날짜 계산을 KST 기준으로 통일하고 useApi(url, initialData) 커스텀 훅으로 SSR 초기 데이터를 9개 페이지에 일관 적용해 해결했습니다.

**[문제 해결] N+1 쿼리 → 병렬 배치 최적화**
관리자 대시보드에서 팀별 통계를 개별 조회하던 N+1 패턴으로 API 응답이 3초 이상 걸렸습니다. 쿼리를 Promise.all 병렬 배치로 재구성하고 복합 인덱스 5개를 추가해 35회 → 9회(74% 감소), 응답 시간 3초 → 0.5초로 개선했습니다.

**[문제 해결] 카카오 인앱브라우저 투표 실패**
카카오톡에서 공유한 링크로 투표 시 세션 쿠키가 전달되지 않아 투표가 실패하는 문제가 있었습니다. 카카오 인앱브라우저의 credential 정책이 원인이었고, fetch에 credentials: "include"를 적용하고 인앱브라우저 감지 시 외부 브라우저 유도 안내를 추가해 해결했습니다.

**[문제 해결] 투표 Optimistic UI + 롤백**
참석 투표 시 API 응답을 기다리는 동안 UI가 멈추는 느낌이 있었습니다. 클릭 즉시 투표 수를 낙관적으로 반영하고, API 실패 시 이전 상태로 롤백하는 패턴을 구현했습니다. 300ms 연속 클릭 방지와 성공 후 백그라운드 refetch로 최종 정합성을 보장합니다.

**[기타 구현]**
- 선호 포지션 우선순위 큐 + 쿼터 균등 분배 알고리즘으로 라인업 20분 → 3초 자동 생성
- VAPID 기반 Web Push 직접 구현 (Service Worker, 구독 관리, Vercel Cron 리마인더)
- useApi 글로벌 캐싱 (60초 TTL, mutation 시 자동 무효화)으로 불필요한 API 호출 제거
- 자체전(A팀 vs B팀) 별도 스코어보드, 팀 편성, 전적 반영 토글 등 복합 기능
- Vitest 615개 단위 테스트, Vercel 자동 배포, PWA(홈 화면 추가 + 오프라인)

기술 스택: Next.js 16, React 19, TypeScript, Supabase(PostgreSQL), TailwindCSS 4, Vercel, Web Push(VAPID)

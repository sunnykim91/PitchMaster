# PitchMaster 만들면서 기술적으로 까다로웠던 것들

> PitchMaster 개발기 시리즈 2편 — 6가지 삽질 정리

[1편](https://velog.io/@sunnykim91/pitchmaster-1)에서 왜 만들었는지 풀었으니 이번엔 만들면서 좀 까다로웠던 것들 6가지. 혼자 만드는 사이드 프로젝트라 삽질하면 그대로 내 시간이 날아가는 구조라, 기억에 남는 게 많습니다.

---

## 1. 은행마다 다른 회비 데이터 — 정규화 + AI Vision 전환

회비 정산 기능이 가장 까다로웠어요.

총무들이 올리는 데이터가 세 종류:
- **통장 스크린샷** (카카오뱅크/토스/국민/신한 ...)
- **엑셀 파일** (은행마다 포맷 천차만별)
- **수기 입력**

문제는 은행마다 포맷이 다르다는 거예요.

```
카카오뱅크:  03.12  이원빈  +30,000원  866,840원
국민은행:    2025.03.12 14:23  이원빈  30000  KRW  866840
토스:         3월 12일 오후 2:23  이원빈  +30,000  잔액 866,840
```

처음엔 은행별 분기 처리를 했어요. 카카오뱅크면 이 정규식, 국민이면 저 정규식. 근데 **은행 추가될 때마다 코드가 늘어나고 엣지 케이스가 끝없이 튀어나왔습니다.**

### 1차 해결: 정규화 파이프라인

접근을 바꿨어요. 입력이 뭐든 간에 최종적으론 "날짜·금액·이름" 세 개만 뽑으면 되거든요. 중간에 정규화 단계 하나 넣었습니다.

```ts
type BulkRow = {
  date: string;        // YYYY-MM-DD
  amount: number;      // 숫자
  description: string; // "이원빈" or "이원빈 회식비"
  type: "INCOME" | "EXPENSE";
};
```

어떤 포맷이 들어와도 이 형태로 통일. 새 은행 추가 시 파서 하나만 더 만들면 끝.

### 2차 해결: Claude Haiku Vision으로 메인 전환

근데 정규식 한계가 명확했어요. **카카오뱅크 그리드 레이아웃** (날짜·이름·시간·금액·잔액이 각자 다른 줄로 OCR됨) 같은 걸 못 잡았습니다.

```
04.27        ← 날짜
이원빈       ← 이름
13:15        ← 시간
1원          ← 금액
866,840원    ← 잔액
```

이걸 "이름 + 금액"이 한 줄에 있다고 가정한 정규식이 못 잡아요. **시간이 누락된 행은 "정보 부족"으로 분류**되고, 사용자가 매번 직접 확인해야 했습니다.

해결: **Claude Haiku Vision API로 메인 경로 전환.** Vision LLM은 줄 구조에 의존 안 하고 시각적으로 거래를 이해합니다.

```ts
// 기존: Clova OCR → 정규식 파싱 (그리드 레이아웃 약함)
// 변경: Claude Vision → JSON 직접 받음 (시각적 이해)
const aiResult = await runAiOcr(file);
if (aiResult && aiResult.rows.length > 0) {
  // 정상 영역으로 직행
}
// AI 실패 시에만 Clova 폴백
```

비용 안전망: **이미지 해시 캐시** (같은 캡처 재호출 무료) + **레이트리밋** (user 20건/일, team 100건/일).

월 30분 걸리던 회비 정산이 지금은 **스크린샷 한 장 → 30초**.

## 2. 9개 페이지에서 반복되던 데이터 처리 — useApi 훅 추출

서비스가 커지면서 짜증 나기 시작했어요.

모든 페이지에서 같은 패턴 반복:
1. 서버에서 초기 데이터 fetch → 화면 그리기
2. 사용자 액션 → API 호출 → 데이터 갱신
3. 로딩 / 에러 / 캐시 처리

대시보드, 회비, 경기 목록, 기록, 게시판... **9개 페이지에서 같은 코드를 9번 작성**.

페이지마다 각자 구현하다 보니:
- 어떤 페이지는 로딩 처리 까먹음
- 어떤 페이지는 에러 처리 빠짐
- **서버에서 가져온 데이터 vs 클라이언트에서 갱신한 데이터 불일치 버그** (Next.js 서버 컴포넌트 + 클라이언트 useState 충돌)

### 해결: `useApi` 커스텀 훅

```ts
const { data, loading, error, refetch } = useApi<DashboardData>(
  '/api/dashboard',
  { initialData: serverData }  // 서버 fetch 결과 inject
);
```

여기에 추가 기능 넣었어요:
- **60초 캐시**: 같은 URL 60초 안에 재요청 → 캐시 반환
- **자동 invalidation**: 데이터 수정 시 관련 캐시 자동 무효화
- **백그라운드 갱신**: 앱이 백그라운드로 갔다 돌아올 때 30초+ 지났으면 자동 refetch

이 훅 하나 만들고 9개 페이지가 거의 같은 모양이 됐어요. 새 페이지도 useApi 한 줄. **데이터 불일치 버그도 사라짐.**

## 3. 라인업 자동 배치 알고리즘 — 3단계 분해

사용자가 가장 좋아하는 기능인데 만들 때 머리 가장 많이 썼어요.

15명을 4쿼터에 나누는 건 단순해 보이지만 조건이 많습니다:
- 선수마다 **선호 포지션** 여러 개 (`["CM", "CAM"]` 등)
- **출전 시간 균등 분배** (15명 × 11명/쿼터 → 누구는 3쿼터, 누구는 4쿼터)
- **감독 지정** 포지션 고정 (`coach_positions`)
- **GK** 별도 관리 (1쿼터 1명)
- **용병** (실력 불확실)
- **하프 쿼터** (전반만 / 후반만)

### 해결: 3단계 분해

```
[1단계] 쿼터 수 분배
  전체 슬롯(11×4=44) ÷ 선수(15) = 평균 2.93 쿼터
  → 누구는 3쿼터, 누구는 2쿼터로 round-robin

[2단계] 쿼터별 슬롯 채우기
  남은 슬롯 가장 많은 쿼터부터 그리디
  하프 쿼터는 2명 묶어서 한 슬롯 공유 (1쿼터 전반/후반)

[3단계] 포지션 매칭
  선호 포지션 우선
  이전 쿼터에 같은 포지션만 한 선수 → 다른 포지션으로 로테이션
```

20분 걸리던 라인업이 **버튼 한 번 → 3초**.

흥미로운 건 사용자 반응이에요. **사람이 짜면 "왜 나만 수비야"** 하는데, **알고리즘이 짜면 그대로 받아들입니다.** "공정하다"는 느낌이 핵심이더라고요.

## 4. 카카오톡 인앱브라우저 쿠키 이슈 — `credentials: "include"` 한 줄

이건 진짜 삽질이었어요.

PitchMaster 링크를 카톡으로 공유하면 팀원이 누를 때 **카톡 인앱브라우저**로 열려요. 일반 크롬·사파리랑 동작이 살짝 달라요.

증상: **로그인은 되는데 투표 버튼 누르면 "로그인이 필요합니다"**.

원인 찾는데 한참 걸렸어요. **인앱브라우저는 fetch 시 쿠키가 자동으로 안 날아가더라고요.**

```ts
// Before
const res = await fetch("/api/votes", { method: "POST", body });

// After — 한 줄 추가로 해결
const res = await fetch("/api/votes", {
  method: "POST",
  body,
  credentials: "include",  // ← 이거
});
```

이거 한 줄인데 찾는 데 며칠 걸렸습니다.

추가 처리:
- **인앱브라우저 감지** (UserAgent에 `KAKAOTALK` 포함) → "외부 브라우저로 열기" 버튼 노출
- **Android**: Intent Scheme으로 크롬 강제 호출
- **iOS**: Safari `target="_blank"`로 새 탭

PWA 설치 / 푸시 알림이 인앱에선 안 되니 우회로가 필수였어요.

## 5. Optimistic UI — 투표 버튼이 멈춘 것처럼 보이는 문제

기술적으론 어렵지 않은데 UX 측면에서 중요했어요.

투표 버튼 누르면 API 응답까지 0.5~1초. 그 사이 버튼이 무반응이면 **"눌린 건가?" 하고 또 누릅니다.** → 같은 투표 두 번 들어감.

### 해결: 즉시 반영 + 실패 시 롤백

```ts
const handleVote = async (vote: VoteType) => {
  // 1) 즉시 화면에 반영 (낙관적 업데이트)
  setVoteCounts(prev => ({ ...prev, [vote]: prev[vote] + 1 }));

  try {
    const res = await fetch("/api/votes", { /* ... */ });
    if (!res.ok) throw new Error();
  } catch {
    // 2) 실패 시 롤백
    setVoteCounts(prev => ({ ...prev, [vote]: prev[vote] - 1 }));
    showToast("투표 실패", "error");
  }
};
```

연속 클릭 방지(0.3초 디바운스)도 같이 넣었어요. 적용 후 만족도 확 올라갔습니다.

> **"동작하는 코드"** 와 **"사용자가 자연스럽게 쓰는 코드"** 는 다르다 — 이때 좀 느꼈어요.

## 6. AI 기능 5개 — Haiku 4.5로 비용·품질 균형

지금까지 추가한 AI 기능 5종:

| 기능 | 모델 | 트리거 |
|------|------|------|
| **선수 시그니처 카드** | Haiku 4.5 | 선수 페이지 첫 진입 |
| **경기 후기 자동 생성** | Haiku 4.5 | 경기 종료 후 일지 탭 |
| **AI 코치 분석** | Haiku 4.5 | 전술판에 배치 후 |
| **AI Full Plan** (자동편성) | Haiku 4.5 | 자동 편성 빌더 |
| **OCR 거래 파싱** | Haiku 4.5 Vision | 회비 일괄 등록 |

### 비용 통제 패턴

5개 다 다음 안전망 적용:

1. **이미지·텍스트 해시 캐시** — 같은 입력 재호출 시 토큰 0
2. **레이트리밋** — feature별 user/team 일별 한도
3. **24h TTL 결과 캐시** — 팀 통계 등 매번 재계산 X
4. **source 필드** (`"ai" | "rule" | "error"`) — 실패·룰 폴백은 한도 차감 X

### 시스템 프롬프트 hallucination 방지

AI 코치 분석에서 가장 신경 썼어요:

```
규칙 0) 포메이션 이름 hallucination 금지
- formationName 값 그대로만 사용
- placement 숫자 집계해서 "4-6-0" 같은 창작 금지

규칙 1) playerRotation 슬롯 기준으로만 역할 서술
- 1쿼터 placement만 보고 고정 서술 X

규칙 2) positionChangers 의 선수는 쿼터 변화 반드시 서술
- "1쿼터 DM → 2쿼터 ST"

규칙 3) 용병(isGuest=true)에 단정적 표현 금지
규칙 4) 상대팀 이력 없으면 "첫 대결"로 명시
```

LLM이 그럴듯하게 거짓말하는 거 막는 게 진짜 일이었어요.

## 정리

혼자 만들면서 느낀 건, **처음부터 완벽하게 만드는 건 불가능**하다는 거예요.

- 회비 파싱: 케이스별 분기 → 정규화 → AI Vision 메인
- 데이터 로딩: 페이지마다 따로 → useApi 훅 통합
- 라인업: 손으로 짜기 → 알고리즘 → AI Full Plan
- 인앱브라우저: 못 잡던 버그 → credentials 한 줄 + 우회 가이드
- 투표 UX: API 대기 → Optimistic UI
- AI: 단순 호출 → 캐시 + 레이트리밋 + 프롬프트 가드

매주 실제로 쓰면서 "이거 좀 불편한데" 하나씩 고친 결과예요.

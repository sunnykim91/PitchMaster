만들면서 기술적으로 까다로웠던 것들


1편에서 왜 만들었는지를 썼는데, 이번에는 만들면서 좀 까다로웠던 부분들을 얘기해보려고 합니다.

혼자 만드는 사이드 프로젝트다 보니 누가 도와주는 것도 아니고, 삽질하면 그대로 내 시간이 날아가는 구조예요. 그래서 기억에 남는 것들이 꽤 있습니다.


## 은행마다 다른 회비 데이터

회비 정산 기능이 제일 까다로웠어요.

총무들이 올리는 데이터가 세 종류예요. 통장 스크린샷, 엑셀 파일, 수기 입력. 문제는 이게 은행마다 포맷이 다 다르다는 거예요.

카카오뱅크는 날짜가 "03.12"로 나오고, 국민은행 엑셀은 "2025.03.12"로 나오고, 토스는 또 다르게 생겼어요. OCR로 스크린샷을 읽으면 글자가 잘려서 나오는 경우도 있고요.

처음에는 은행별로 분기 처리를 했어요. 카카오뱅크면 이 로직, 국민이면 저 로직. 근데 은행이 추가될 때마다 코드가 늘어나고, 엣지 케이스가 계속 튀어나왔습니다.

결국 접근을 바꿨어요.

입력이 뭐든 간에, 최종적으로는 "날짜, 금액, 이름" 이 세 가지만 뽑아내면 되거든요. 그래서 중간에 정규화 단계를 하나 넣었어요. 어떤 포맷으로 들어오든 날짜는 YYYY-MM-DD로, 금액은 숫자로, 이름은 문자열로 통일하는 거예요.

그렇게 하니까 새로운 은행 포맷이 들어와도 파서만 하나 추가하면 되고, 뒤쪽 매칭 로직은 건드릴 필요가 없었습니다.

매칭도 좀 신경 썼어요. OCR로 읽은 이름이 "김민수"인데 팀원 목록에 "김민수"가 있으면 바로 연결해주는 건데, 이게 간단해 보여도 실제로는 이름 앞뒤에 공백이 붙어 있거나, "김민수 FC"처럼 뒤에 뭔가 붙어 있는 경우가 있어요. 이런 것들을 포함 매칭으로 처리했습니다.

중복 거래도 걸러야 했어요. 같은 날짜에 같은 금액이 두 번 들어오면 중복인지 진짜 두 번 입금한 건지 판단해야 하는데, DB에 이미 있는 레코드랑 날짜+금액+유형을 비교해서 필터링하는 식으로 해결했어요.

월 30분 걸리던 회비 정산이 지금은 스크린샷 한 장 올리면 30초면 끝나요.


## 9개 페이지에서 반복되던 데이터 처리

서비스가 커지면서 짜증났던 게 있어요.

모든 페이지에서 같은 패턴이 반복돼요. 서버에서 초기 데이터를 가져와서 화면을 그리고, 사용자가 뭔가 하면 API를 호출해서 데이터를 갱신하는 거. 대시보드, 회비, 경기 목록, 기록, 게시판... 전부 다요.

처음에는 페이지마다 각자 구현했어요. 그러다 보니 어떤 페이지는 로딩 상태 처리를 까먹고, 어떤 페이지는 에러 처리가 빠지고, 서버에서 가져온 데이터랑 클라이언트에서 갱신한 데이터가 안 맞는 버그가 여기저기서 터졌어요.

특히 서버/클라이언트 데이터 불일치가 골치 아팠습니다. Next.js에서 서버 컴포넌트로 데이터를 먼저 그려주잖아요. 근데 클라이언트에서 같은 데이터를 다시 가져오면 미묘하게 다른 경우가 있어요. 날짜 계산 같은 게 서버 시간이랑 클라이언트 시간이 다르면 결과가 달라지거든요.

결국 useApi라는 커스텀 훅을 하나 만들었어요.

이 훅이 하는 일은 간단해요. URL이랑 서버에서 받은 초기 데이터를 넘기면, 초기 데이터로 일단 화면을 그리고, 이후에 필요할 때 API를 호출해서 갱신해주는 거예요.

여기에 몇 가지를 더 넣었어요. 같은 URL을 60초 안에 또 요청하면 캐시에서 바로 돌려주고, 데이터를 수정하면 관련 캐시를 자동으로 날려주고, 앱을 백그라운드로 보냈다가 다시 돌아오면 30초 이상 지났으면 자동으로 갱신해주는 거.

이 훅 하나 만들고 나서 9개 페이지의 데이터 로딩 코드가 거의 다 같은 모양이 됐어요. 새 페이지를 만들 때도 useApi 한 줄이면 끝이고, 데이터 불일치 버그도 사라졌습니다.


## 라인업 자동 배치

사용자들이 제일 좋아하는 기능인데, 만들 때 제일 머리를 많이 썼어요.

얼핏 보면 단순해 보여요. 15명을 4쿼터에 나눠 넣으면 되니까. 근데 조건이 많아요.

선수마다 선호 포지션이 있어요. 누구는 공격만 하고 싶고, 누구는 수비도 괜찮고. 쿼터별로 출전 시간을 균등하게 맞춰야 하는데, 15명이 4쿼터에 11명씩 들어가면 누군가는 3쿼터만 뛰고 누군가는 4쿼터 다 뛰게 돼요. 이걸 공정하게 나눠야 합니다.

거기에 감독이 특정 선수를 특정 포지션에 고정할 수도 있고, GK는 따로 관리해야 하고, 용병도 있고, 하프 쿼터(전반만 뛰는 경우)도 있어요.

처음에는 복잡하게 느껴졌는데, 정리하니까 3단계였어요.

첫 번째, 각 선수가 몇 쿼터를 뛰어야 하는지 계산해요. 전체 슬롯 수를 선수 수로 나눠서 평균을 구하고, 누군가는 한 쿼터 더, 누군가는 한 쿼터 덜 뛰게 분배합니다.

두 번째, 쿼터별로 선수를 배정해요. 남은 슬롯이 가장 많은 쿼터부터 채워나가는 식으로요. 하프 쿼터 선수는 2명씩 묶어서 한 슬롯을 공유하게 했어요.

세 번째, 배정된 선수를 실제 포지션에 넣어요. 선호 포지션을 먼저 보고, 이전 쿼터에서 같은 포지션만 계속 한 선수가 있으면 다른 포지션으로 돌려줍니다.

이렇게 정리하니까 20분 걸리던 라인업이 버튼 한 번에 3초면 나와요. "공정하다"는 느낌을 주는 게 중요한데, 알고리즘으로 하니까 아무도 불만을 안 해요. 사람이 짜면 "왜 나만 수비야" 하지만, 기계가 짜면 받아들이더라고요.


## 카카오톡 인앱브라우저

이건 진짜 삽질이었어요.

PitchMaster 링크를 카카오톡으로 공유하잖아요. 팀원이 그 링크를 누르면 카카오톡 안에 있는 브라우저로 열려요. 이게 인앱브라우저인데, 일반 크롬이나 사파리랑 동작이 좀 달라요.

제일 큰 문제는 투표가 안 되는 거였어요. 로그인은 됐는데 투표 버튼을 누르면 "로그인이 필요합니다"라고 나와요. 분명 로그인했는데.

원인을 찾아보니 인앱브라우저에서 쿠키 처리가 다르더라고요. API 요청할 때 쿠키가 같이 안 날아가는 거예요.

모든 API 요청에 credentials: "include"를 넣어서 해결했어요. 이거 한 줄인데 찾는 데 꽤 걸렸습니다.

그리고 인앱브라우저에서는 PWA 설치(홈 화면 추가)도 안 되고, 푸시 알림도 안 돼요. 그래서 인앱브라우저를 감지하면 "브라우저에서 열기" 버튼을 보여주는 식으로 처리했어요.

안드로이드에서는 Intent Scheme이라는 걸 써서 크롬으로 강제로 열어주고, iOS에서는 사파리로 새 탭을 열어주는 방식이에요. UserAgent 문자열에 "KAKAOTALK"이 있으면 인앱으로 판단합니다.


## 투표 버튼이 멈춘 것처럼 보이는 문제

이건 기술적으로 어렵진 않은데, 사용자 경험 측면에서 중요했어요.

투표 버튼을 누르면 API 호출이 가는데, 응답이 올 때까지 0.5~1초 정도 걸려요. 그 사이에 버튼이 아무 반응이 없으면 "눌린 건가?" 하고 또 누르게 돼요.

그래서 버튼을 누르는 순간 바로 화면에 반영해버렸어요. 참석을 눌렀으면 참석 수를 즉시 +1 해주는 거죠. 그리고 API가 성공하면 그대로 두고, 실패하면 다시 원래대로 돌려놓는 방식이에요.

이걸 보통 Optimistic UI라고 하는데, 적용하니까 투표가 즉각적으로 반응하는 느낌이 들어서 만족도가 확 올라갔어요. 연속 클릭 방지도 넣어서 0.3초 안에 같은 버튼을 못 누르게 했고요.

"동작하는 코드"랑 "사용자가 자연스럽게 쓸 수 있는 코드"는 다르다는 걸 이때 좀 느꼈어요.


## 정리하면

혼자 만들면서 느낀 건, 처음부터 완벽하게 만드는 건 불가능하다는 거예요.

회비 파싱도 처음엔 케이스별 분기였다가 정규화 파이프라인으로 바뀌었고, 데이터 로딩도 페이지마다 따로 구현하다가 useApi로 모였고, 라인업도 손으로 짜다가 알고리즘으로 바뀌었어요.

매주 실제로 쓰면서 "이거 좀 불편한데" 하는 걸 하나씩 고치다 보니 이렇게 됐습니다. 다음 편에서는 0명에서 300명까지 사용자를 어떻게 모았는지를 쓸 예정이에요.
